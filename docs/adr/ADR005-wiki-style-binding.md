# ADR 005 — file-stem as default wiki-link completion style

## Context

Wiki-link completion refers to how the server populates the `insertText` of a completion item when the user types `[[` and begins entering a document reference. Three styles were considered:

**`file-stem`** — The completion text is the file name without its extension. For a file located at `vault/projects/my-note.md`, the completion text is `my-note`. This is the style Obsidian itself uses in its native auto-complete popup. It matches what a user sees when they open a file in the Obsidian file explorer and matches the canonical display name used in the graph view. Obsidian's own link resolution defaults to shortest unique file stem, making `file-stem` style links the most Obsidian-native choice.

**`title-slug`** — The completion text is derived from the first heading (`# H1`) of the target document, slugified into a wiki-link-safe form. For a file whose first heading is `## Data Flow Diagram`, the completion text might be `data-flow-diagram` or `Data Flow Diagram` depending on slugification strategy. This style is preferred in Zettelkasten workflows where the note ID and the note title are intentionally coupled. marksman defaults to this style. A significant number of Obsidian users do not write explicit H1 headings, which makes title-slug unreliable for them.

**`file-path-stem`** — The completion text is the full path relative to the vault root, without the `.md` extension. For `vault/projects/my-note.md`, this would be `projects/my-note`. This style is unambiguous and supports disambiguation between files with the same name in different folders. It is, however, verbose and unfamiliar to users who are accustomed to Obsidian's shortest-path resolution.

Obsidian's own link-resolution algorithm uses shortest unique file stem by default; it only falls back to path-qualified links when two files share a stem. This means `file-stem` style is what native Obsidian links look like in practice.

All three styles must be supported because users migrating from marksman (`title-slug`) or managing vaults where folder structure matters (`file-path-stem`) need to be able to opt in. The question is which style is the default.

## Decision

Default completion style is **`file-stem`**, controlled by configuration key `completion.wiki.style` (values: `"file-stem"`, `"title-slug"`, `"file-path-stem"`; default: `"file-stem"`).

All completion items generated by the wiki-link completion provider respect the active `completion.wiki.style`. All rename refactoring (see [[features/rename]]) that updates existing links must produce output consistent with the active style — renaming a file updates the stem portion of every affected wiki-link, not the path.

When `title-slug` is active and a document has no H1 heading, the server falls back to `file-stem` for that document and logs a debug-level warning.

## Consequences

**Positive:**

- Consistent with Obsidian's own native auto-complete and link resolution behaviour. Users who are accustomed to Obsidian's interface will see familiar completion items.
- Eliminates the most common friction point: a user types `[[my` and gets `my-note`, which is exactly what Obsidian would produce.
- Rename refactoring is simpler under `file-stem` style because the completion text is directly derived from the filename, and renaming the file is all that is required to update the stem.

**Negative:**

- `title-slug` users (particularly those migrating from marksman) must explicitly set `completion.wiki.style = "title-slug"` in `.flavor-grenade.toml` to get their preferred behaviour.
- In vaults with many files sharing the same stem across different folders, `file-stem` produces ambiguous completions. The completions provider must handle this by appending a folder qualifier in the completion label (but not in `insertText`) so the user can distinguish candidates.

**Neutral:**

- The style is a configuration value, not a hard-coded behaviour. Changing the default in a future release is non-breaking for users who have set `completion.wiki.style` explicitly.
